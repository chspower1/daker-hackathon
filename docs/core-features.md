# 핵심 기능 명세

## 1. 홈

- 입력: 사용자가 랜딩 화면에 진입한다 -> 동작: 서비스 소개, 핵심 메시지, 주요 CTA를 보여준다 -> 결과: 서비스가 무엇을 제공하는지 한눈에 이해할 수 있다.
- 입력: 사용자가 `해커톤 탐색하기`, `캠프로 이동`, 랭킹 관련 CTA를 누른다 -> 동작: 각각 `/hackathons`, `/camp`, `/rankings` 로 이동한다 -> 결과: 원하는 기능 페이지로 바로 진입한다.

## 2. 해커톤 목록

- 입력: 사용자가 `/hackathons` 에 진입한다 -> 동작: `readHackathons()` 결과를 읽고, 저장소를 사용할 수 없으면 오류 상태를 표시한다 -> 결과: 공개 해커톤 카드 목록 또는 오류 상태가 일관되게 표시된다.
- 입력: 사용자가 뷰 모드(카드/테이블) 토글을 클릭한다 -> 동작: 목록의 표시 방식을 카드 또는 테이블 형태로 전환한다. 테이블 모드에서는 주요 표시 컬럼(제목, 상태, 시작/종료일) 헤더를 클릭하여 오름차순/내림차순 정렬이 가능하다 -> 결과: 사용자가 선호하는 레이아웃과 정렬 기준으로 정보를 빠르게 파악할 수 있다.
- 입력: 사용자가 목록 상단의 키워드 검색어 창에 입력한다 -> 동작: 제목, 태그 등 표시되는 주요 텍스트를 기준으로 해커톤을 필터링한다 -> 결과: 기존 상태 및 태그 필터와 결합하여 더욱 정교한 조건으로 해커톤을 검색할 수 있다.
- 입력: 사용자가 상태 필터 버튼을 선택한다 -> 동작: `upcoming`, `ongoing`, `ended`, `all` 기준으로 목록을 다시 계산하며 각 상태별 현재 필터링된 개수를 표시한다 -> 결과: 조건에 맞는 해커톤만 남고 예상 개수를 미리 알 수 있다.
- 입력: 사용자가 태그를 여러 개 선택한다 -> 동작: 선택된 태그 중 하나라도 포함하는 해커톤만 남기면서 추가 태그 선택 UI를 계속 유지한다 -> 결과: OR 조건으로 여러 태그를 이어서 탐색할 수 있다.
- 입력: 사용자가 태그 버튼을 선택한다 -> 동작: 다중 선택(Toggle)이 가능하며, 선택된 태그 중 하나라도 포함된(OR 조건) 해커톤만 필터링한다. 이미 선택된 태그를 다시 누르면 해제되고, 'All tags'를 누르면 모든 태그 선택이 해제된다 -> 결과: 여러 관심 분야에 맞는 해커톤을 유연하게 탐색할 수 있다.
- 입력: 사용자가 태그 검색 입력창에 검색어를 입력한다 -> 동작: 태그 목록을 이름 기준으로 즉시 필터링하고, 태그가 많을 때도 사이드바 전체가 아니라 태그 영역 내부에서만 스크롤되도록 유지한다. 검색 결과가 없으면 태그 전용 빈 메시지를 보여준다 -> 결과: 긴 태그 목록에서도 사이드바 구조를 유지한 채 원하는 태그를 빠르게 찾을 수 있다.
- 입력: 필터 결과가 없다 -> 동작: 필터 전용 빈 상태 메시지를 표시한다 -> 결과: 사용자는 다른 조합으로 다시 시도할 수 있다.
- 입력: 사용자가 해커톤 카드를 클릭한다 -> 동작: 카드의 `links.detail` 경로로 이동한다 -> 결과: 선택한 해커톤 상세 페이지가 열린다.
- 입력: 목록 읽기에 실패한다 -> 동작: 오류 상태 화면을 표시한다 -> 결과: 사용자는 목록을 읽을 수 없음을 즉시 인지한다.

## 3. 해커톤 상세

- 입력: 사용자가 `/hackathons/[slug]` 에 진입한다 -> 동작: 요약, 상세, 팀, 제출, 리더보드 데이터를 slug 기준으로 읽는다 -> 결과: 하나의 화면에서 해커톤 전체 맥락을 확인할 수 있다.
- 입력: 일부 섹션 데이터가 없다 -> 동작: 해당 섹션만 `EmptyState` 로 대체한다 -> 결과: 페이지 전체는 유지되고 비어 있는 정보만 분리해서 보여준다.
- 입력: 사용자가 상단 섹션 탭을 누른다 -> 동작: `overview`, `info`, `eval`, `schedule`, `prize`, `teams`, `submit`, `leaderboard` 8개 섹션 중 하나로 앵커 이동하며, 클릭 시에만 URL hash 가 업데이트된다 -> 결과: 필요한 섹션을 빠르게 탐색할 수 있다.
- 입력: 사용자가 페이지를 스크롤한다 -> 동작: sticky 헤더 바로 아래 임계값(Threshold)을 기준으로 현재 섹션을 판정(Scroll-spy)하여 사이드 목차(TOC)의 해당 항목을 강조(Highlight)하고 `aria-current="location"` 을 부여한다. 스크롤 중에는 URL hash 를 자동으로 변경하지 않는다 -> 결과: 현재 읽고 있는 위치를 시각적으로 즉시 파악할 수 있다.
- 입력: 모바일 환경에서 사용자가 스크롤한다 -> 동작: 활성화된 섹션 항목이 가로 스크롤 목차 영역의 중심에 오도록 자동으로 위치를 보정한다 -> 결과: 좁은 화면에서도 현재 위치를 놓치지 않고 확인할 수 있다.

### 팀

- 입력: 사용자가 `팀` 섹션을 본다 -> 동작: 현재 slug 와 연결된 모집글을 `teams` 저장소 기준으로 읽고, 없으면 공개 시드 데이터를 사용한다 -> 결과: 현재 해커톤의 팀 현황이 카드 목록으로 표시된다.
- 입력: 사용자가 `캠프로 이동` 버튼을 누른다 -> 동작: `/camp?hackathon={slug}` 로 이동한다 -> 결과: 같은 해커톤 기준의 모집글 화면으로 연결된다.

### Submit

- 입력: 로컬 프로필이 없는 사용자가 `Submit` 섹션에 진입한다 -> 동작: 프로필 생성 폼과 안내 메시지를 보여준다 -> 결과: 제출 전에 닉네임 기반 로컬 프로필을 만들 수 있다.
- 입력: 사용자가 닉네임을 입력하고 프로필을 만든다 -> 동작: `createLocalProfile()` 과 `saveLocalProfile()` 를 실행한다 -> 결과: 현재 브라우저에서 제출 가능한 프로필이 준비된다.
- 입력: 사용자가 팀 이름, 산출물 링크, 메모를 입력하고 `Save Draft` 를 누른다 -> 동작: 제출 데이터를 `draft` 상태로 `submissions` 에 저장한다 -> 결과: 임시 저장 상태와 마지막 수정 시간이 화면에 표시된다.
- 입력: 사용자가 같은 폼에서 `Final Submit` 을 누른다 -> 동작: `submissions` 를 `submitted` 로 갱신하고 `leaderboards` 도 같은 팀명 기준으로 함께 갱신한다 -> 결과: 제출 완료 메시지와 함께 리더보드 반영 상태를 확인할 수 있다.
- 입력: 리더보드 저장이 실패한다 -> 동작: 직전 제출 변경에 대한 rollback 을 시도하고 오류 메시지를 보여준다 -> 결과: 제출과 리더보드 데이터 불일치를 최소화하고 현재 저장 상태를 다시 반영한다.

### Leaderboard

- 입력: 사용자가 `Leaderboard` 섹션을 본다 -> 동작: 제출 여부, 순위, 점수, 제출 시각, 산출물 링크를 정렬해서 보여준다 -> 결과: 공개 가능한 결과만 기준에 맞게 비교할 수 있다.
- 입력: 제출되지 않은 항목이 있다 -> 동작: 순위보다 뒤로 배치하고 `submittedAt` 이 없는 상태로 보여준다 -> 결과: 실제 제출팀과 미제출팀이 구분된다.

## 4. 캠프

- 입력: 사용자가 `/camp` 에 진입한다 -> 동작: `teams`, `hackathons`, `localProfile` 을 읽어 목록과 작성 가능 상태를 준비한다 -> 결과: 모집글 탐색 화면이 열린다.
- 입력: 사용자가 뷰 모드(카드/테이블) 토글을 클릭한다 -> 동작: 카드 뷰나 테이블 뷰로 표시 방식을 변경하며, 테이블 모드에서는 각 컬럼별 정렬(오름차순/내림차순)을 지원한다 -> 결과: 한눈에 여러 팀을 비교하거나 상세한 카드로 살펴볼 수 있다.
- 입력: 사용자가 목록 상단의 키워드 검색어 창에 입력한다 -> 동작: 팀명, 소개, 팀 성향, 역할, 해커톤 이름, 작성자 등 팀 모집 관련 텍스트에서 키워드를 검색해 결과를 좁힌다 -> 결과: 단순 상태/태그 필터 외에도 자유로운 텍스트 검색이 가능하다.
- 입력: URL 에 `hackathon` 쿼리가 있다 -> 동작: 해당 slug 와 일치하는 모집글만 필터링한다 -> 결과: 특정 해커톤 관련 모집글만 집중해서 볼 수 있다.
- 입력: 사용자가 상태 필터를 선택한다 -> 동작: `isOpen` 기준으로 `모집 중`, `모집 마감`, `전체` 목록과 각 상태별 현재 필터링 건수를 다시 계산한다 -> 결과: 원하는 모집 상태의 글만 빠르게 볼 수 있다.
- 입력: 사용자가 태그를 선택하거나 검색한다 -> 동작: `모집 분야` 와 `팀 성향` 카테고리의 태그를 검색하고, 선택된 태그 중 하나라도 포함된 모집글만 남기며 추가 태그 선택 UI를 계속 유지한다 -> 결과: OR 조건으로 여러 태그를 이어서 좁혀 볼 수 있다.
- 입력: 상태/태그 필터 결과가 없다 -> 동작: 필터 전용 빈 상태 메시지를 보여준다 -> 결과: 사용자는 다른 조합으로 다시 시도할 수 있다.
- 입력: 사용자가 `모집글 작성` 버튼을 누른다 -> 동작: 작성 마법사 모달을 연다. 프로필이 없으면 첫 단계로 프로필 생성 단계를 먼저 보여준다 -> 결과: 프로필 준비 여부에 따라 다음 단계로의 진입이 나뉜다.
- 입력: 사용자가 마법사에서 닉네임을 입력해 프로필을 만든다 -> 동작: 로컬 프로필을 저장하고 다음 단계로 넘어간다 -> 결과: 같은 브라우저에서 모집글 작성이 가능해진다.
- 입력: 사용자가 마법사를 통해 `teamName`, `intro`, `lookingFor`, `teamStyle`, `contactUrl`, `hackathonSlug`, `isOpen` 을 입력하고 제출한다 -> 동작: 새 `TeamPost` 를 만들어 `teams` 에 저장한다. `contactUrl` 은 공개 가능한 외부 링크로 취급하며 유효한 `http`/`https` 값만 실제 연락 링크로 유지한다 -> 결과: 모달이 닫히고 새 모집글이 즉시 목록 맨 앞에 나타나며 새로고침 후에도 유지된다.

## 5. 글로벌 랭킹

- 입력: 사용자가 `/rankings` 에 진입한다 -> 동작: `readRankings()` 로 데이터를 읽고 `rank` 기준으로 정렬한다 -> 결과: 랭킹 표가 순위순으로 표시된다.
- 입력: `rankings` 저장소가 갱신되거나 사용자가 탭으로 다시 돌아온다 -> 동작: 화면이 최신 값을 다시 읽고 동일 데이터면 상태 갱신을 생략한다 -> 결과: 불필요한 재렌더링 없이 최신 랭킹을 유지한다.
- 입력: 랭킹 데이터가 없다 -> 동작: 빈 상태 메시지를 보여준다 -> 결과: 사용자는 아직 시즌 데이터가 없음을 이해한다.
- 입력: 랭킹 읽기에 실패한다 -> 동작: 오류 상태와 재시도 동작을 보여준다 -> 결과: 접속 불가 상황을 별도 상태로 처리한다.

## 6. 공통 기능 규칙

- 입력: 사용자가 헤더의 `로그인` 버튼을 누르고 프로필 ID를 입력한다 -> 동작: 입력한 ID를 `id` 와 `nickname` 으로 사용하는 모의 `localProfile` 을 생성하고 로컬 스토리지에 저장한다 -> 결과: 로그인 상태로 전환되며 프로필 ID와 `로그아웃` 버튼이 헤더에 나타난다.
- 입력: 사용자가 헤더의 `로그아웃` 버튼을 누른다 -> 동작: 로컬 스토리지의 `localProfile` 을 `null` 로 저장해 로그인 상태를 해제한다 -> 결과: 로그아웃 상태로 전환되며 프로필이 필요한 화면은 즉시 권한 요청 뷰로 바뀐다.
- 입력: 사용자가 읽기 전용 페이지에 접근한다 -> 동작: 프로필 없이도 데이터를 읽을 수 있게 한다 -> 결과: 탐색과 확인 기능은 누구나 사용할 수 있다.
- 입력: 사용자가 쓰기 동작을 시도한다 -> 동작: `localProfile` 존재 여부를 먼저 확인한다 -> 결과: 프로필 없는 상태에서는 생성 유도 후에만 작성/제출이 가능하다.
- 입력: 사용자가 언어 전환을 선택한다 -> 동작: locale 과 사전을 바꾸고 주요 문구와 메타데이터를 다시 동기화한다 -> 결과: 한국어/영어 전환과 새로고침 유지가 동작한다.
- 입력: 사용자가 상단 헤더의 테마 토글을 누른다 -> 동작: 라이트/다크 테마를 전환하고 현재 선택을 `localStorage.app-theme` 에 저장한다 -> 결과: 주요 화면이 즉시 다크모드로 전환되고 새로고침 후에도 유지된다.
- 입력: 브라우저 저장소에 접근할 수 없거나 저장된 테마 값을 읽을 수 없다 -> 동작: 초기 렌더는 시스템 `prefers-color-scheme` 값을 기준으로 테마를 결정한다 -> 결과: 저장소 예외 상황에서도 초기 테마가 시스템 선호와 크게 어긋나지 않는다.
- 입력: 화면에 민감 정보나 내부 식별자가 포함될 가능성이 있다 -> 동작: 공개 가능한 값만 렌더링한다 -> 결과: 서비스 문서 기준의 공개 범위를 지킨다.
