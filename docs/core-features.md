# 핵심 기능 명세

## 1. 홈

- 입력: 사용자가 랜딩 화면에 진입한다 -> 동작: 서비스 소개, 핵심 메시지, 주요 CTA를 보여준다 -> 결과: 서비스가 무엇을 제공하는지 한눈에 이해할 수 있다.
- 입력: 사용자가 `해커톤 탐색하기`, `팀 찾기`, 랭킹 관련 CTA를 누른다 -> 동작: 각각 `/hackathons`, `/camp`, `/rankings` 로 이동한다 -> 결과: 원하는 기능 페이지로 바로 진입한다.

## 2. 해커톤 목록

- 입력: 사용자가 `/hackathons` 에 진입한다 -> 동작: `readHackathons()` 를 읽고 비어 있으면 시드 데이터를 fallback 으로 사용한다 -> 결과: 공개 해커톤 카드 목록이 표시된다.
- 입력: 사용자가 상태 필터 버튼을 선택한다 -> 동작: `upcoming`, `ongoing`, `ended`, `all` 기준으로 목록을 다시 계산한다 -> 결과: 조건에 맞는 해커톤만 남는다.
- 입력: 사용자가 태그 버튼을 선택한다 -> 동작: 선택한 태그를 포함한 항목만 필터링한다 -> 결과: 관심 분야에 맞는 해커톤만 볼 수 있다.
- 입력: 필터 결과가 없다 -> 동작: 필터 전용 빈 상태 메시지를 표시한다 -> 결과: 사용자는 다른 조합으로 다시 시도할 수 있다.
- 입력: 사용자가 해커톤 카드를 클릭한다 -> 동작: 카드의 `links.detail` 경로로 이동한다 -> 결과: 선택한 해커톤 상세 페이지가 열린다.
- 입력: 목록 읽기에 실패한다 -> 동작: 오류 상태 화면을 표시한다 -> 결과: 사용자는 목록을 읽을 수 없음을 즉시 인지한다.

## 3. 해커톤 상세

- 입력: 사용자가 `/hackathons/[slug]` 에 진입한다 -> 동작: 요약, 상세, 팀, 제출, 리더보드 데이터를 slug 기준으로 읽는다 -> 결과: 하나의 화면에서 해커톤 전체 맥락을 확인할 수 있다.
- 입력: 일부 섹션 데이터가 없다 -> 동작: 해당 섹션만 `EmptyState` 로 대체한다 -> 결과: 페이지 전체는 유지되고 비어 있는 정보만 분리해서 보여준다.
- 입력: 사용자가 상단 섹션 탭을 누른다 -> 동작: `overview`, `info`, `eval`, `schedule`, `prize`, `teams`, `submit`, `leaderboard` 앵커로 이동한다 -> 결과: 필요한 섹션을 빠르게 탐색할 수 있다.

### Teams

- 입력: 사용자가 `Teams` 섹션을 본다 -> 동작: 현재 slug 와 연결된 모집글을 `teams` 저장소 기준으로 읽고, 없으면 공개 시드 데이터를 사용한다 -> 결과: 현재 해커톤의 팀 현황이 카드 목록으로 표시된다.
- 입력: 사용자가 `Go to Camp` 버튼을 누른다 -> 동작: `/camp?hackathon={slug}` 로 이동한다 -> 결과: 같은 해커톤 기준의 모집글 화면으로 연결된다.

### Submit

- 입력: 로컬 프로필이 없는 사용자가 `Submit` 섹션에 진입한다 -> 동작: 프로필 생성 폼과 안내 메시지를 보여준다 -> 결과: 제출 전에 닉네임 기반 로컬 프로필을 만들 수 있다.
- 입력: 사용자가 닉네임을 입력하고 프로필을 만든다 -> 동작: `createLocalProfile()` 과 `saveLocalProfile()` 를 실행한다 -> 결과: 현재 브라우저에서 제출 가능한 프로필이 준비된다.
- 입력: 사용자가 팀 이름, 산출물 링크, 메모를 입력하고 `Save Draft` 를 누른다 -> 동작: 제출 데이터를 `draft` 상태로 `submissions` 에 저장한다 -> 결과: 임시 저장 상태와 마지막 수정 시간이 화면에 표시된다.
- 입력: 사용자가 같은 폼에서 `Final Submit` 을 누른다 -> 동작: `submissions` 를 `submitted` 로 갱신하고 `leaderboards` 도 같은 팀명 기준으로 함께 갱신한다 -> 결과: 제출 완료 메시지와 함께 리더보드 반영 상태를 확인할 수 있다.
- 입력: 리더보드 저장이 실패한다 -> 동작: 직전 제출 변경을 rollback 하고 오류 메시지를 보여준다 -> 결과: 제출과 리더보드 데이터의 불일치를 남기지 않는다.

### Leaderboard

- 입력: 사용자가 `Leaderboard` 섹션을 본다 -> 동작: 제출 여부, 순위, 점수, 제출 시각, 산출물 링크를 정렬해서 보여준다 -> 결과: 공개 가능한 결과만 기준에 맞게 비교할 수 있다.
- 입력: 제출되지 않은 항목이 있다 -> 동작: 순위보다 뒤로 배치하고 `submittedAt` 이 없는 상태로 보여준다 -> 결과: 실제 제출팀과 미제출팀이 구분된다.

## 4. 캠프

- 입력: 사용자가 `/camp` 에 진입한다 -> 동작: `teams`, `hackathons`, `localProfile` 을 읽어 목록과 작성 가능 상태를 준비한다 -> 결과: 모집글 탐색 화면이 열린다.
- 입력: URL 에 `hackathon` 쿼리가 있다 -> 동작: 해당 slug 와 일치하는 모집글만 필터링한다 -> 결과: 특정 해커톤 관련 모집글만 집중해서 볼 수 있다.
- 입력: 사용자가 `모집글 작성` 버튼을 누른다 -> 동작: 작성 마법사 모달을 연다. 프로필이 없으면 첫 단계로 프로필 생성 단계를 먼저 보여준다 -> 결과: 프로필 준비 여부에 따라 다음 단계로의 진입이 나뉜다.
- 입력: 사용자가 마법사에서 닉네임을 입력해 프로필을 만든다 -> 동작: 로컬 프로필을 저장하고 다음 단계로 넘어간다 -> 결과: 같은 브라우저에서 모집글 작성이 가능해진다.
- 입력: 사용자가 마법사를 통해 `teamName`, `intro`, `lookingFor`, `contactUrl`, `hackathonSlug`, `isOpen` 을 입력하고 제출한다 -> 동작: 새 `TeamPost` 를 만들어 `teams` 에 저장한다 -> 결과: 모달이 닫히고 새 모집글이 즉시 목록 맨 앞에 나타나며 새로고침 후에도 유지된다.

## 5. 글로벌 랭킹

- 입력: 사용자가 `/rankings` 에 진입한다 -> 동작: `readRankings()` 로 데이터를 읽고 `rank` 기준으로 정렬한다 -> 결과: 랭킹 표가 순위순으로 표시된다.
- 입력: `rankings` 저장소가 바뀌거나 주기적 재조회가 발생한다 -> 동작: 화면이 최신 값을 다시 읽고 동일 데이터면 상태 갱신을 생략한다 -> 결과: 불필요한 재렌더링 없이 최신 랭킹을 유지한다.
- 입력: 랭킹 데이터가 없다 -> 동작: 빈 상태 메시지를 보여준다 -> 결과: 사용자는 아직 시즌 데이터가 없음을 이해한다.
- 입력: 랭킹 읽기에 실패한다 -> 동작: 오류 상태와 재시도 동작을 보여준다 -> 결과: 접속 불가 상황을 별도 상태로 처리한다.

## 6. 공통 기능 규칙

- 입력: 사용자가 읽기 전용 페이지에 접근한다 -> 동작: 프로필 없이도 데이터를 읽을 수 있게 한다 -> 결과: 탐색과 확인 기능은 누구나 사용할 수 있다.
- 입력: 사용자가 쓰기 동작을 시도한다 -> 동작: `localProfile` 존재 여부를 먼저 확인한다 -> 결과: 프로필 없는 상태에서는 생성 유도 후에만 작성/제출이 가능하다.
- 입력: 사용자가 언어 전환을 선택한다 -> 동작: locale 과 사전을 바꾸고 주요 문구와 메타데이터를 다시 동기화한다 -> 결과: 한국어/영어 전환과 새로고침 유지가 동작한다.
- 입력: 화면에 민감 정보나 내부 식별자가 포함될 가능성이 있다 -> 동작: 공개 가능한 값만 렌더링한다 -> 결과: 서비스 문서 기준의 공개 범위를 지킨다.
